# AWS 과제 범위
# 과제 1
> 목표: “면접에서 개발은 깊게 몰라도, AWS 뒷단 구조와 트래픽 흐름, 장애 대응, 모니터링은 막힘없이 설명할 수 있는 상태로 만드는 것.”
> 

핵심은 **반복 숙달 + 설명 가능성**

### 과제1 요구사항

| 항목 | 반드시 설명해야 할 내용 |
| --- | --- |
| 서비스 배경 | 왜 이 서비스를 만들었는가 |
| 목적 | 어떤 문제를 해결하는가 |
| 컨셉추얼 아키텍처 | 사용자, 프론트, 백엔드, DB, 외부 서비스 흐름 |
| AWS 인프라 아키텍처 | VPC, Subnet, ALB, EC2/EKS, RDS, S3, IAM, Security Group |
| Terraform | RDS, S3, EC2, LB를 코드로 생성 |
| 네트워크 트래픽 | 사용자가 요청했을 때 어디를 거쳐 어디로 가는가 |
| EKS | 컨테이너 이미지가 어떻게 배포되고 서비스되는가 |
| 모니터링 | LGTM 또는 Prometheus/Grafana/Loki 기반으로 무엇을 관측하는가 |
| 장애 대응 | 어디서 장애가 나면 무엇을 먼저 보는가 |
| 비용 | 어떤 리소스가 비용을 만들고 어떻게 줄일 수 있는가 |
- **중요한 핵심:** **면접에서 그림 없이 말로 설명 가능한 수준** (멋진 구조X)


### 산출물 가이드
- 새 서비스(주제·도메인): 신뢰할 수 있는 인생 의사결정 가이드 서비스
- 산출물
    - 컨셉추얼 아키
    - AWS 인프라 아키
    - Terraform 리소스 목록
    - 장애 대응 설명(장애 시, 제일 먼저 확인할 것)
    
# AI 활용 원칙(과제 수행 원칙)
| 방식 | 평가 |
| --- | --- |
| AI로 코드 생성 후 이해 못함 | 위험 |
| AI로 초안 생성 후 구조 이해·수정·검증 | 좋음 |
| **AI 없이 0 to 100 구현** | **강점** |
| AI 사용 내역을 README에 정리 | 오히려 투명하고 좋음 |

**절대 원칙**: 초안 생성 이상의 작업 시, 작업 중지 및 사용자에게 한번 더 물어보고 진행할지 여/부 결정.
